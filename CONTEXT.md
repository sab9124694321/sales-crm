# SZB CRM — Контекст проекта

## О проекте
CRM-система для отдела продаж эквайринга Сбербанка.
Управление продажами и сопровождение эквайринга.
Стек: PHP + SQLite + GigaChat AI (Сбер)
Хостинг: Docker на VPS (szb-sales.ru)

## Команда
- Работаем отделами продаж эквайринга Сбербанка
- "Охотники" — посторонние люди (внешние агенты), приводящие лиды
- Мы — внутренние менеджеры

## Ограничения безопасности (критично!)
| Ограничение | Почему | Как обходим |
|-------------|--------|-------------|
| Нельзя хранить данные клиентов на внешних серверах | Запрет безопасности Сбера | Используем почту, планируем интеграции |
| Нет доступа к микрофону с рабочего ноутбука | Политика безопасности | Анализ голоса — только на Android |
| Не работает кнопка отправки почты на корпоративных iPad | Техническое ограничение | Ищем обходные пути |
| Telegram заблокирован в России | Регуляторное ограничение | Планировали автоотчёты в Макс и на почту |

## Иерархия ролей (кто кого контролирует)

```
Термен (terman)
    └── Начальник управления (territory_head)
            ├── Начальник Отдела продаж УБР (head)
            │       └── Менеджер (manager / ubr_middle)
            └── Начальник ММБ (mmb_tp_head)
                    └── Менеджер ММБ (mmb_manager)
```

Связь в БД: `users.manager_id → users.id`
- `manager_id` у manager/ubr_middle → id руководителя head
- `manager_id` у mmb_manager → id руководителя mmb_tp_head
- `manager_id` у head/mmb_tp_head → id руководителя territory_head
- `territory_id` у всех → привязка к территории для фильтров термена

### Smart-форма (5 обязательных полей)
1. Что беспокоит клиента? (pain_point)
2. Какие возражения звучали? (objection + objection_text)
3. Что договорились? (next_step)
4. Кто принимает решение? (decision_maker) — роль, не ФИО
5. Когда следующий контакт? (next_call_date + next_call_time)

### Фрод-скор (0-100)
| Баллы | За что |
|-------|--------|
| 0-20 | Длина комментария |
| 0-20 | Заполнены все обязательные поля |
| 0-15 | Детализация возражения |
| 0-15 | Конкретность договорённостей |
| 0-10 | Дата следующего контакта |
| 0-10 | Свободный комментарий |
| -30 | Односложные комментарии ("да", "ок", "дозвон") |
| -25 | Копипаст (похожесть > 80% с предыдущими) |
| -15 | Подозрительные фразы (≥3 из списка) |

### Результаты фрод-скора
| Скор | Действие |
|------|----------|
| ≥ 70 | ✅ Звонок верифицирован |
| 40-69 | ⚠️ Требует внимания |
| < 40 | 🚨 На контроль РОПа |

### РОП-контроль (rop_control.php)

#### Что видит каждая роль
| Роль | Видит |
|------|-------|
| manager / ubr_middle / mmb_manager | Только свою статистику: план/день, задачи, звонки, на контроле, подтверждено |
| head | Таблицу по manager/ubr_middle + итоги + **блок контроля задач** |
| mmb_tp_head | Таблицу по mmb_manager + итоги + **блок контроля задач** |
| territory_head / terman | Иерархию по территориям: Территория → Начальник → Команда (разворачивается пошагово) |
| admin | Всё + блок контроля всех задач |

#### Блок контроля задач (для head / mmb_tp_head / admin)
- Задачи со скором < 60 попадают в `rop_control_queue`
- Фильтр по статусу: На проверке / Подтверждено / Отклонено / Перепрозвон / Все
- Карточка задачи: номер, фрод-скор, менеджер, территория, комментарий
- Кнопки: ✅ Подтвердить / ❌ Отклонить / 📞 Перепрозвон
- Комментарий РОПа **обязателен** при отклонении/перепрозвоне
- При подтверждении — задача → «Подтверждена"
- При отклонении — задача → «Назначена» (возвращается в пул)
- При перепрозвоне — задача → «На контроле РОП»

#### Иерархия территорий (для territory_head / terman)
- **Территория** → кнопка «👥 Показать руководителей» → разворачиваются начальники
- **Начальник** → кнопка «👥 Команда» → разворачиваются сотрудники
- Итоги по каждому уровню: план, задачи, звонки, на контроле, подтверждено, отклонено, перепрозвон
- В CSV-выгрузке — вся иерархия сразу

#### Две кнопки CSV
- 📥 **Статистика CSV** — план, задачи, звонки, на контроле, подтверждено, отклонено, перепрозвон
- 📋 **Задачи CSV** — номер задачи, менеджер, статус задачи, статус звонка, комментарий, дата

### План звонков (единая логика)
```php
// Из таблицы plans, поле calls_plan за текущий месяц
$stmt = $pdo->prepare("SELECT calls_plan FROM plans WHERE tabel_number = ? AND period = strftime('%Y-%m', 'now')");
$stmt->execute([$tabel]);
$month_plan = $stmt->fetchColumn() ?: 350;  // Дефолт 350
$work_days = 22;
$daily_plan = ceil($month_plan / $work_days);  // Дневной план
```
Используется в: `dashboard.php`, `calls.php`, `rop_control.php`.

## Безопасный деплой (04.07.2026)

### Проблема
Каждый раз после git pull или git reset --hard таблицы пропадали из базы sales.db.

### Причина
sales.db был в индексе git. git reset --hard перезаписывает ВСЕ файлы в папке.

### Решение
1. sales.db удален из индекса git (`git rm --cached sales.db`)
2. sales.db добавлен в `.gitignore`
3. Создан скрипт `/usr/local/bin/safe-crm-update.sh` на сервере
4. Cron изменен: вместо git pull теперь запускается safe-crm-update.sh

### Что делает safe-crm-update.sh
| Шаг | Зачем |
|-----|-------|
| `cp sales.db sales.db.BACKUP_AUTO` | Бэкап перед обновлением |
| `git stash` | Скрывает sales.db от git |
| `git pull origin main` | Подтягивает код |
| `git stash pop` | Восстанавливает sales.db |
| `test -f sales.db` | Если git стер — восстановить из бэкапа |
| `chown www-data:www-data` | Правильные права |
| `chmod 664` | Правильные права |
| `sqlite3 ...` | Проверка таблиц, автосоздание |

### Cron на сервере
```
*/5 * * * * docker exec sales-crm bash -c "/usr/local/bin/safe-crm-update.sh" >> /var/log/crm-auto-update.log 2>&1
```
